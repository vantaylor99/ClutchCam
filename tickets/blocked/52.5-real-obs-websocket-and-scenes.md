The repo-side OBS preflight is ready, but this workstation's current OBS
configuration is not ready for real hook testing.

What I found on July 1, 2026:

- OBS Studio is installed at `C:\Program Files\obs-studio\bin\64bit\obs64.exe`.
- The `obs-websocket` plugin loads, but the local OBS WebSocket config has
  `server_enabled=false`, so port `4455` is closed during normal startup.
- The current scene collection contains `Main`, `4k Webcam Only`, and
  `Mic Cam Scene`.
- The required ClutchCam scene names are absent:
  `Quad View`, `Player 1 Fullscreen`, `Player 2 Fullscreen`,
  `Player 3 Fullscreen`, and `Player 4 Fullscreen`.

What happens if we do nothing:

The real OBS preflight and app startup will fail before any manual scene control
or real-stream transcription test can run. That is the correct safe behavior,
because the app must not switch to scenes that do not exist.

Recommended default:

Open OBS manually, enable `Tools > WebSocket Server Settings`, keep
authentication enabled, and create or rename the five required scenes exactly as
listed above. Then run the preflight from `ai-stream-director/`:

```powershell
$env:DRY_RUN_OBS="false"
$env:OBS_HOST="127.0.0.1"
$env:OBS_PORT="4455"
$env:OBS_PASSWORD="<local OBS WebSocket password>"
python scripts/smoke_obs_connection.py
```

Alternative:

Approve an automated temporary OBS scene-setup script that creates the five
required scene names for hook testing. This is reversible, but it mutates the
local OBS scene collection and should be a human-approved operation rather than
something an agent does silently.

Once unblocked, run the manual-scene-control ticket before any real stream or
transcription test.
