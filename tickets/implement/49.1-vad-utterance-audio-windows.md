Implement the first non-chunked source mode as local voice activity detection before provider requests. The source should keep the existing FFmpeg extractor and existing `Transcriber` adapters, but instead of sending every fixed chunk directly to the provider, it should assemble speech windows around detected speech starts and silence pauses, write bounded local WAV request files, and transcribe those utterance windows.

Design decision: voice activity detection runs before provider requests for this first pass. Provider streaming APIs are deliberately not implemented here; the source boundary from the prerequisite ticket leaves room for them later. Pre-request local detection gives the app utterance-level finals while reducing request volume during silence, and it works with both existing transcription request modes because it still produces local `AudioInputRef` values.

Implementation shape:

- Support only normalized mono WAV extraction for the VAD source. If the configured audio extraction container, codec, channel count, or sample rate cannot be read safely with the standard `wave` module, startup should fail clearly and the operator can return to `TRANSCRIPTION_SOURCE_MODE=chunked`.
- Read stable completed WAV chunks in media order using `CompletedAudioChunkDiscovery` or a small shared helper derived from it. Preserve `_infer_chunk_start_seconds(...)` media timestamps.
- Use a simple, deterministic local speech detector over PCM frames with configurable frame length, minimum speech duration, minimum silence duration before finalization, leading padding, trailing padding, and maximum utterance duration.
- Write request WAV files under the audio extraction directory, for example in `_vad/<stream_id>/`, with media-derived names. These files are inputs to the existing `Transcriber.transcribe(AudioInputRef)` call.
- Build utterance `AudioInputRef` values with `starts_at_seconds`, `duration_seconds`, sample rate, channels, codec, and `emit_from_seconds` when padding or reconnect overlap could duplicate earlier speech.
- Map provider segment timestamps back onto media time through the existing transcription normalization. Text-only provider responses may use the utterance window duration as they do today, but timestamped responses should be preferred for trigger timing.
- Emit only provider final events as final switching candidates. The local detector may emit diagnostic partials only if it can do so without changing switch behavior; otherwise leave partial diagnostics for a later ticket.

Long speech must not block forever. When speech continues beyond the configured maximum utterance duration, finalize a window at a safe frame boundary with trailing padding, then immediately continue a new active utterance. This preserves bounded request sizes and keeps `TranscriptRouter` responsible for assembling nearby final events into trigger candidates.

Duplicate handling should be conservative. If reconnects, extractor restarts, or VAD padding cause a new request to include audio that ended at or before the last emitted media time for that stream, suppress events whose `end_time_seconds` are not newer than the stream's last emitted final event. Do not deduplicate only by text because repeated phrases can be valid gameplay speech.

## Edge cases & interactions

- Speech that starts before an extractor restart and ends after it should produce either one later finalized utterance or two timestamp-ordered final events, not an exception or duplicate trigger.
- Silence-only chunks should not call the provider, and the source should still advance its processed chunk state.
- Background game audio and crosstalk can produce false starts; minimum speech duration and silence thresholds should keep short bursts from creating requests.
- Long monologues must finalize before they exceed max utterance duration, request timeout assumptions, or router candidate limits.
- Missing chunks, empty chunks, unreadable WAV files, and incompatible WAV parameters should fail per chunk or per source with diagnostic output, not crash the orchestrator thread without surfacing the startup/runtime error.
- If the provider returns text-only responses, the emitted event should cover the utterance request window; if it returns segment or word timestamps, use those timestamps to keep trigger time precise.
- Padding must not re-emit text from previous finalized speech when timestamped provider segments make suppression possible.
- Backpressure is still handled by the sink. The VAD source should not spin rapidly on silence or retry a failed request in a tight loop.
- Cleanup should remove stale generated VAD request WAV files that are no longer needed, following the existing overlap cleanup pattern.

## Tests and validation

- Unit tests should build small synthetic WAV chunks with silence, speech-like PCM frames, silence gaps, and long speech to prove utterance windows start and end at expected media timestamps.
- Tests should prove silence-only input does not invoke the transcriber.
- Tests should prove trailing silence finalizes an utterance and long speech finalizes at the maximum duration.
- Tests should prove timestamped provider responses are shifted onto media time and overlap/padding-only events are suppressed.
- Tests should prove text-only provider responses still emit bounded final events using the utterance request duration.
- Tests should cover unreadable or incompatible WAV chunks and verify the worker reports failures without losing later valid chunks where possible.
- Run focused transcription extraction/runtime/worker tests and the full Python unit suite from `ai-stream-director/`.
