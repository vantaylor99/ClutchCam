After the orchestrator can consume live transcription events in process, the
local Linux Compose defaults need to avoid running two audio extraction paths at
the same time. The current Compose file gives `transcription-worker` the
`local-linux` profile and also mounts the audio extraction directory into the
`orchestrator` service. That is fine while the worker only emits standalone
JSONL diagnostics, but it would duplicate FFmpeg extraction and Faster-Whisper
requests once the orchestrator owns live transcription.

Keep the standalone `transcription-worker` service as an explicit diagnostic
profile and healthcheck target. The integrated local Linux path should run live
transcription from the orchestrator when the new runtime flag is enabled, using
the same `TRANSCRIPTION_*`, `AUDIO_EXTRACT_*`, `AUDIO_INPUT_URL_*`,
`INGEST_API_URL`, and shared `/dev/shm/clutchcam-audio` settings. The
orchestrator service should receive the transcription and audio-extraction env
needed by `AppConfig`; the worker service should not be part of the default
integrated profile set at the same time.

Expected behavior:

- The default documented integrated local stack starts one transcription path,
  not both the orchestrator live source and the standalone worker.
- Operators can still run `COMPOSE_PROFILES=transcription-worker` or the
  explicit `transcription-worker` service for JSONL diagnostics and healthcheck
  validation.
- Orchestrator dry-run OBS mode can be combined with live transcription so the
  first runtime validation does not require real OBS playback.
- The optional `local-transcription` Faster-Whisper service remains separate
  from the default local Linux profile; operators opt into it only when the host
  has enough CPU/GPU capacity.
- Buffered OBS media-source injection remains out of scope. This ticket should
  not wire `BufferBackedSwitcher` or `MediaSourceOutputSwitcher` into `main()`.

Validation should be static/unit-level unless a human explicitly requests a live
Compose run.
