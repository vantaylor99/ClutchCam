text, but every accepted line stopped at the local transcript prefilter. Because
`evaluate_accepted_transcript(...)` returns `prefilter_rejected` when
`TranscriptTriggerPrefilter.classify(...)` returns no candidate signal, the AI
director never evaluated several plausible gameplay moments.

Hypothesis: the prefilter itself is working as designed, but the default
`hype_phrases` list is too narrow for live team-play callouts. The fix should
add a small, tested set of gaming callout phrases without turning the prefilter
into a broad keyword detector that escalates routine filler or repeated
hallucinated outro text.

Implementation guidance:

- Keep `TranscriptTriggerPrefilter` cheap and deterministic; this should remain
  a local phrase gate before model escalation, not a model-like classifier.
- Add only the smallest default phrase set needed for the known eval misses.
  Favor phrases that are specific enough to gameplay moments:
  "that was nasty", "behind you", "good work", "triple", and "legend".
- Treat "help" carefully. A lone "help" was repeated in the eval and can be a
  meaningful team-play callout, but the current `min_text_characters = 6` means
  a single word "help" cannot match without broadening the minimum length for
  every transcript. Prefer a phrase-specific or context-safe solution if this is
  included; do not lower the global minimum in a way that admits short filler
  such as "ok", "no", "hi", or punctuation-only noise.
- Preserve duplicate suppression across streams and across nearby repeated
  phrases. Repeated "behind you" or repeated "help" inside the duplicate window
  should not cause repeated AI calls.
- Preserve rejection of the existing filler list and routine outro-style text
  that contains no configured trigger phrase.

Expected behavior after implementation:

- A final transcript event containing "that was nasty" produces a transcript
  `HypeSignal` with confidence at or above the configured minimum.
- A final transcript event containing "behind you" produces one local candidate
  signal, while a nearby repeat on another stream is rejected as a duplicate.
- The runtime event pipeline calls `ai_director.decide(...)` when a newly
  accepted runtime transcript contains one of the added phrases.
- Routine filler such as "yeah", "thanks", and punctuation-only noise still
  returns no local signal.
- Existing hype phrases, discovery phrases, gameplay phrases, reference-time
  handling, disabled configuration behavior, and duplicate suppression continue
  to pass.

## Edge cases & interactions

- Short phrases: do not solve "help" by reducing the global minimum text length
  unless tests prove short filler remains rejected.
- Substring matching: the current matcher uses normalized substring search, so
  avoid adding ambiguous fragments that match ordinary words unexpectedly.
- Duplicate detection: `_same_hype_phrase(...)` suppresses nearby events that
  match the same configured phrase even when the full transcript differs; tests
  should cover the newly added phrases.
- Runtime pipeline: `process_transcript_event(...)` routes accepted final
  events before prefiltering, then `evaluate_accepted_transcript(...)` skips the
  model when no candidate signal exists. Add coverage at this boundary so the
  change is not only tested at the service class level.
