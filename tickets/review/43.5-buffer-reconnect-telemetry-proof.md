Reconnect acceptance now uses data-plane advancement as the primary
machine-verifiable criterion. FFmpeg child exit/start logs remain useful
diagnostics when the child actually exits or restarts, but they are no longer
required for acceptance when SRS keeps the RTMP consumer process alive across
bounded publishers.

Implementation notes for review:

- `compose_generated_ingest_checkpoint.py` now supports `--reconnect-proof`,
  `--reconnect-gap-seconds`, `GENERATED_INGEST_RECONNECT_PROOF`, and
  `GENERATED_INGEST_RECONNECT_GAP_SECONDS`.
- The checkpoint report always includes a `reconnect` section. It is `not_run`
  by default and becomes `passed` or `failed` only when reconnect proof is
  enabled.
- Reconnect proof runs a second bounded publish after the first ready buffer,
  waits for every requested stream to remain clip-ready, and requires each
  latest segment sequence to advance beyond the first ready-buffer sequence.
- The proof also compares the `buffer-worker` Compose service identity before
  and after the second publish, preferring the Compose JSON `ID` field and
  falling back to container name only when an ID is unavailable.
- Compose service-state parsing now preserves container IDs for the identity
  check.
- Buffer supervision tests now make the process telemetry contract explicit:
  when a child exits and is restarted, logs include `buffer_ffmpeg_exited` and
  later `buffer_ffmpeg_started` entries.
- The local Linux Compose runbook documents segment advancement as the
  reconnect acceptance gate and describes FFmpeg exit/start logs as diagnostic
  evidence only when a child actually exits or restarts.
