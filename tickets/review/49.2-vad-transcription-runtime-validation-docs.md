Implemented the VAD runtime-validation and documentation pass.

What changed:

- Documented `TRANSCRIPTION_SOURCE_MODE=chunked` as the default/fallback and `TRANSCRIPTION_SOURCE_MODE=vad-utterance` as an optional local speech-pause source in the architecture doc, README, and local Linux runbook.
- Distinguished source mode from `TRANSCRIPTION_REQUEST_MODE`, which still only controls the provider upload/API shape.
- Documented operator comparison guidance for missed trigger phrases, duplicate triggers, speech-end-to-accepted-transcript latency, host/backend resource use, provider request count/duration, and backend cost.
- Added runbook examples for comparing `chunked` and `vad-utterance` against the same fixture or rehearsal stream.
- Kept the latency soak harness offline and fixture-based; no Docker, FFmpeg, OBS, GPU inference, or network dependency was added.
- Added configured JSONL diagnostic fields for the standalone transcription worker: `transcription_source_mode` and `transcription_request_mode`.
- Added startup guidance so incompatible VAD extraction settings point operators back to `TRANSCRIPTION_SOURCE_MODE=chunked`.
- Added runtime coverage proving a final VAD-shaped `TranscriptEvent` uses the existing `TranscriptRouter`, prefilter, AI director, and scheduler path.
- Preserved existing partial-event rejection coverage in `LiveTranscriptQueueSink` and `process_transcript_event(...)`.
- Updated the roadmap to reflect the optional VAD source path and move this ticket into review.
